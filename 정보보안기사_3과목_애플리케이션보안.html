FTP — Active/Passive 모드, ftpusers, xferlog

FTP는 파일 전송 프로토콜이야. Active/Passive 포트 동작 방식과, 접근 제어 파일인 ftpusers, 로그 파일인 xferlog가 보안 핵심이야.
FTP의 종류 필기+실기
종류설명
ftpID 및 Password 인증을 수행하고 TCP 프로토콜을 사용하여 사용자의 데이터를 송수신
tftp인증과정 없이 UDP 기반으로 데이터를 빠르게 송수신. 69번 포트 사용. 인증이 없어서 보안에 취약
sftp전송구간에 암호화 기법을 사용하여 기밀성을 제공. SSH 기반으로 동작
📋
ftpusers 파일: 특정 사용자에 대해서 ftp 접근을 차단할 수 있어. 여기에 등록된 계정은 FTP 로그인 불가.
xferlog 파일: FTP의 서비스 로그파일 중 하나. 업로드·다운로드 기록 확인 가능.
Active mode vs Passive mode 필기+실기

FTP는 제어 채널(21번 포트)과 데이터 채널을 따로 사용해. 데이터 채널을 누가 열어주느냐에 따라 Active/Passive로 나뉘어.

전송 방식동작 방식
Active mode ① FTP Client에서 FTP Server 21번 포트로 접속
② FTP Client는 FTP Server 20번 포트로 데이터를 전송
→ 서버가 클라이언트에게 능동적으로 데이터 채널 연결
Passive mode ① FTP Client에서 FTP Server 21번 포트로 접속
② FTP Server가 FTP Client로 데이터 송수신을 위해서 1,024~65,535 범위의 Random 포트를 선택
→ 서버가 수동적으로 클라이언트의 연결을 기다림
💡
Active = 서버가 능동적(Active)으로 클라이언트에게 20번 포트로 연결 시도
Passive = 서버가 수동적(Passive)으로 Random 포트를 열고 클라이언트 연결을 기다림

이메일 보안 — SMTP, PGP, PEM, S/MIME, 스팸메일 차단

이메일은 피싱·스팸·악성코드 유포의 주요 경로야. 프로토콜 구조와 암호화 방식, 스팸 차단 기법을 이해하면 실제 보안 설정과도 연결돼.
SMTP 구성요소 필기+실기
구성요소역할
MTA (Mail Transfer Agent)메일을 전송하는 메일서버. 이메일을 라우팅하고 다른 MTA로 전달
MDA (Mail Delivery Agent)수신 측에 고용된 우체부 역할. MTA에게 받은 메일을 사용자에게 전달
MUA (Mail User Agent)사용자들이 사용하는 클라이언트 애플리케이션 (Outlook, Thunderbird 등)
PGP (Pretty Good Privacy) 필기+실기
🔐
PGP 암호화 과정 순서 (암기 필수)
압축 → 세션키 생성 → 메시지 암호화 → 세션키 암호화 → 전자서명 → 전송 → 수신 및 복호화
PGP 서비스사용 알고리즘
전자서명DSS/SHA 또는 RSA/SHA로 전자서명이 가능
메시지 암호화CAST-128, IDEA, 3DES로 메시지 암호화
1회용 세션키 생성Diffie-Hellman 혹은 RSA로 키 분배
이메일 호환RADIX-64로 바이너리를 ACS Code로 변환
세그먼테이션메시지 최대 사이즈를 제한
PEM과 S/MIME 필기+실기

📧 PEM (Privacy Enhanced Mail)

  • 중앙집중화된 키를 인증
  • 구현이 어렵고 높은 보안성을 제공
  • 군대, 은행에서 사용

📧 S/MIME

  • 인터넷 MIME 메시지에 전자서명과 함께 암호화를 더한 프로토콜
  • RSA 암호를 사용
  • CA(인증기관)로부터 X.509 인증서를 받아야 함
스팸메일 차단 기법 필기+실기
기법RBL (Real Time Blocking List)SPF (Sender Policy Framework)
방식 KISA의 RBL 서버에 특정 IP를 등록하고 차단 발신자: 자신의 메일서버 정보와 정책을 나타내는 SPF 레코드를 해당 DNS에 등록.
수신자: 이메일 수신 시 DNS에 등록된 SPF 레코드를 확인하여 해당 이메일에 표시된 발송 IP와 비교하고 수신 여부를 결정

DNS 보안 — 쿼리 종류, 레코드, named.conf, DNS 공격

DNS는 도메인 이름을 IP 주소로 변환하는 시스템이야. 인터넷의 전화번호부 역할인데, 이걸 공격하면 사용자를 가짜 사이트로 유도할 수 있어서 보안적으로 매우 중요해.
DNS Query의 종류 필기+실기
종류설명
Recursive Query (순환)Local DNS 서버에 Query를 보내 완성된 답을 요청. Local DNS 서버가 다른 서버들을 대신 조회해서 최종 결과를 돌려줌. 사용자 호스트(Stub-Resolver)가 이 방식으로 질의 전송
Iterative Query (반복)Local DNS 서버가 다른 DNS 서버에게 Query를 보내서 답을 요청하며, 외부 도메인에서 개별적인 작업을 통해 정보를 얻어와서 종합해서 알려줌. 순환 Name Server가 Master Name Server에 반복질의 수행
named.conf 파일 설정 필기+실기
설정 항목설명
zone설정하려는 도메인명을 기술
type• master: Primary DNS
• slave: Secondary DNS
allow-transferMaster Name Server에서는 Zone 파일을 수신할 Slave Name Server를 지정. none은 전송하지 않음
allow-recursion순환 쿼리를 제한할 IP를 지정
FQDNFully Qualified Domain Name — 전체 도메인 네임
DNS 레코드 종류 필기+실기
레코드설명
A (Address)호스트 이름을 IPv4 주소로 매핑. 단일 호스트 이름에 IP주소가 여러 개이거나, 동일한 IP에 여러 호스트 이름이 있을 수 있음
AAAA호스트 이름을 IPv6 주소로 매핑
PTR (Pointer)특수 이름이 도메인의 다른 위치를 가리킬 수 있음. 인터넷 주소의 PTR 레코드는 정확히 한 개만 있어야 함 (역방향 조회)
NS (Name Server)도메인에는 해당 이름의 서비스 레코드가 적어도 한 개 이상 있어야 하며 이는 DNS 서버이다
MX (Mail Exchanger)도메인 이름으로 보낸 메일을 받도록 구성되는 호스트 목록을 지정
CNAME (Canonical Name)호스트의 다른 이름(별칭)을 정의하는 데 사용
SOA (Start of Authority)도메인에 대한 권한을 갖는 서버를 표시. 도메인에서 가장 큰 권한을 부여받은 호스트를 선언
Any (ALL)위의 모든 레코드를 표시
DNS 공격 방법 필기+실기
💡
DNS 증폭(Amplification) 원리
Open DNS Resolver 서버를 이용해서 DNS Query의 Type을 "ANY"로 설정. A, NS, CNAME, AAAA 등의 모든 레코드를 요청하기 때문에 요청한 쿼리 패킷보다 크게 증폭됨.
공격 종류설명
DNS Zone Transfer 공격DNS 서버에 Zone transfer를 계속 시도하여 시스템 자원과 대역폭을 소진하는 DoS 공격 기법. DNS Zone transfer는 Master DNS 서버와 Slave DNS 서버 간에 Zone 파일 동기화를 수행하는 프로토콜
DNS 증폭공격DNS Query 시에 Any 타입을 사용하고 DNS Response 메시지를 피해자에게 전송시키는 공격
DNS Query FloodingDNS Request를 지속적으로 발생시켜 DNS 서버를 공격하는 DoS 공격 기법
DNS Cache PoisoningDNS 서버의 캐시 테이블을 공격하여 DNS Spoofing을 할 수 있도록 하는 공격 → 가짜 사이트로 유도 가능
Reflector AttackDNS 순환 질의 과정을 악용한 DDoS 공격으로 특정 사이트에 트래픽을 유발

데이터베이스 보안 — 위협 요소, 암호화 방법

데이터베이스 보안 위협 요소 필기
위협 요소설명
집합성 (Aggregation)낮은 보안 등급의 정보들을 이용하여 높은 등급의 정보를 알아내는 것
추론 (Inference)보안 등급이 없는 일반 사용자가 보안으로 분류되지 않은 정보에 정당하게 접근하여 기밀 정보를 추출해 내는 행위. 로우 데이터(Raw Data)로부터 민감한 데이터를 유출하는 행위.
보안 대책: 다중인스턴스화(Polyinstantiation)
데이터베이스 암호화 방법 필기+실기
방식설명
Plug In 방식데이터베이스 서버에 별도의 암호화 솔루션을 설치하고 애플리케이션이 데이터 조작어(insert, update, select 등)를 실행하면 암호화하거나 복호화함
API 방식애플리케이션 레벨에서 암호화/복호화를 처리. 애플리케이션 코드 수정 필요
TDE (Transparent Data Encryption)DBMS 자체에서 지원하는 커널기반 암호화. 특정 DBMS에서만 지원하는 것으로 데이터파일을 암호화하는 것

전자상거래 보안 — 전자화폐 프로토콜, SET

전자화폐 프로토콜 필기
프로토콜설명
인출 프로토콜사용자와 은행 간의 프로토콜. 은행이 사용자에게 전자화폐를 발급해주는 절차를 설명하는 프로토콜
지불 프로토콜사용자와 상점 간의 프로토콜. 사용자가 구매 대금으로 자신의 전자화폐를 상점에 지불하는 과정을 설명하는 프로토콜
예치 프로토콜상점과 은행 간의 프로토콜. 상점이 사용자로부터 전자화폐를 은행에 입금시키는 프로토콜
SET (Secure Electronic Transaction) 필기

SET의 핵심기술은 이중서명과 전자봉투야. 카드 소유자, 가맹점, Payment Gateway 3자 간의 안전한 거래를 보장해.

참여자처리 내용
카드 소유자 • 암호화된 구매정보: 상점의 공개키로 암호화
• 암호화된 결제정보: 대칭키로 암호화
• 검증 해시: 구매정보 해시, 결제정보 해시
• 전자서명: 구매자의 개인키로 암호화
• 전자봉투: 암호화된 결제정보, 암호화된 대칭키를 Payment Gateway의 공개키로 암호화
가맹점 (상점) • 상점은 개인키로 암호화된 구매정보를 복호화
• 상점은 구매정보를 해시함수에 입력하고 수신한 해시값과 일치하는지 확인
• 고객의 공개키로 전자서명을 확인
• 전자서명에서 추출한 검증 해시값이 같은지 확인
Payment Gateway • 전자봉투를 자신의 개인키로 복호화
• 대칭키를 획득하고 결제정보를 복호화

💥
OpenSSL에서 발견된 보안 취약점을 하트블리드(HeartBleed) 취약점이라고 함.

보안약점설명
SQL 삽입사용자의 입력값이 SQL 쿼리에 삽입되어 공격자가 쿼리를 조작·공격할 수 있는 보안약점. PreparedStatement를 사용해야 함. SQL Injection, Blind SQL Injection, Union SQL Injection, Mass SQL Injection

부적절한 XML 외부 개체 참조XML 문서의 DTD(Document Type Definition) 문서 내에 XML 외부 엔티티의 처리를 활성화할 때 발생하는 보안약점

HTTP 응답분할사용자의 입력값 등 외부 입력값이 HTTP 응답헤더에 삽입되어 악의적인 코드가 실행되는 보안약점 (CRLF 개인문자 삽입)

FTP 바운스 Attack 필기+실기
💥
익명의 FTP 사용자를 이용해서 공격자 자신의 위치를 감추고 공격 대상의 네트워크 내부시스템의 포트를 스캔한 후에 PORT 명령어를 사용해서 FTP 서버로 하여금 악의의 헤더 부분을 조작한다. 공격자가 원하는 곳으로 Fake 메일 같은 데이터를 만들어 전송할 수 있는 행위이다.

FTP 공격은 대부분 FTP 바운스 Attack이 출제된다.
FTP 공격 종류설명
FTP 버퍼 오버플로우FTP 서비스의 버퍼 오버플로우 취약점 악용
FTP 바운스 Attack익명 FTP를 이용해 위치 감추고 포트 스캔 및 데이터 전송 → 가장 자주 출제
FTP 익명 사용자 취약점인증 없이 접근 가능한 anonymous 계정 악용
FTP 인증 우회인증 절차를 우회하여 접근

구분세션 쿠키 (Session Cookie)지속 쿠키 (Persistent Cookie)
특징 임시 쿠키. 사용자가 웹 브라우저를 종료하면 쿠키 값이 삭제된다 웹 브라우저를 종료해도 쿠키 값이 저장되어 있다. Expires 혹은 Max-Age 파라미터가 없으면 세션 쿠키가 된다

→ Set-Cookie의 경우 HttpOnly를 설정하면 웹 서버에서 쿠키를 사용하지 못하게 한다 ← 이게 오답

전자서명 기능 — 수신자 부인방지는 아님 필기+실기

FTP Passive Mode — 포트 빈칸 정답 필기+실기
💡
FTP의 Passive Mode는 명령 전송을 위해서 (ㄱ) 포트를 시작하고 데이터 전송을 위해서 서버가 (ㄴ) 포트 이상으로 결정한다.

정답: (ㄱ) 21, (ㄴ) 1024
FTP Passive Mode에서 명령 채널 = 21번 포트, 데이터 채널 = 1024번 이상의 랜덤 포트 (1024~65535)

SMTP 구성요소 — MUA/MTA/MDA/MRA 필기+실기

이메일 전송에서 각 구성요소의 역할 — 빈칸 문제로 자주 출제돼.

구성요소역할
MUA (Mail User Agent)사용자들이 사용하는 클라이언트 애플리케이션 (Outlook, Thunderbird 등). 메일 작성·발송·수신
MTA (Mail Transfer Agent)메일을 전송하는 메일서버. 이메일을 라우팅하고 다른 MTA로 전달
MDA (Mail Delivery Agent)수신 측에서 MTA에게 받은 메일을 사용자에게 전달하는 우체부 역할
MRA (Mail Retrieval Agent)사용자가 메일 서버에서 메일을 가져오는 역할. POP3, IMAP 프로토콜 사용
💡
기출 빈칸 정답: (ㄱ) MUA, (ㄴ) MTA, (ㄷ) MDA
순서: 사용자 → MUA(작성) → MTA(전송) → MDA(배달) → 수신자
웹 방화벽(WAF) 주요 기능 필기+실기

웹 방화벽은 IP 주소와 포트 번호로 차단하는 일반 방화벽과 달리, HTTP/HTTPS 트래픽을 분석해서 웹 공격을 차단하는 보안 장비야.

기능설명
URL 단위 필터링URL에 단위를 필터링한다
SQL Injection, XSS, Command Injection 탐지웹 공격 패턴 탐지 및 차단
Cookie/HTML Hidden 필드 및 Parameter 검증쿠키 및 파라미터 값을 검사한다
파일 업로드 제어 및 파일 검사업로드되는 파일 타입 및 내용 검사
⚠️
웹 방화벽 기능이 아닌 것: "IP 주소와 포트 번호를 사용하여 네트워크를 차단하거나 허용하는 기능"
→ 이건 일반 방화벽(Network Firewall)의 기능. 웹 방화벽은 애플리케이션 계층(L7)에서 HTTP 트래픽을 분석하는 것
DNS Spoofing — 가짜 IP로 유도 필기+실기

사용자가 입력한 URL에 대해서 가짜 IP 주소를 전송하여 공격자가 웹사이트로 연결되도록 한다 → DNS Spoofing

💡
DNS 스푸핑(DNS Spoofing)은 네트워크 공격의 한 종류로, 공격자가 DNS 서버의 응답을 가로채거나 조작하여 사용자를 가짜 웹사이트로 유도하는 기법이다.
OTP — 올바르지 않은 설명 필기+실기
⚠️
"OTP는 응용 계층에서 동작하는 것이다" ❌ → 오답!
OTP는 특정 계층에 종속되지 않아. 응용 계층에서 동작한다는 설명은 잘못됨.

OTP의 올바른 설명:
① OTP 전송 계층에서 서버에서 질의 값을 수신해서 응답하여 진행된다 ✅
② 비동기 방식으로 진행된다 ✅
③ 동기식 방식에는 시간 동기식과 이벤트 동기식이 있는 방식으로 진행된다 ✅
④ OTP는 사용할 때마다 새로운 난수를 발생시킨다 ✅
OTP 생성 및 인증 방식 — 캡쳐 방식은 없음 필기+실기
⚠️
OTP 생성 및 인증 방식이 아닌 것 → "캡쳐(CAPTCHA) 방식"
CAPTCHA는 Agent인지 사람의 행위인지 구분하기 위한 것으로 특정 그림의 숫자를 입력하는 것이야. OTP 인증 방식이 아님!

OTP의 올바른 생성·인증 방식:
• 이벤트 동기 방식 ✅
• 시간 동기 방식 ✅
• 질의-응답 방식 (비동기식) ✅
SSO 보안 위협 — 익명 로그인은 아님 필기+실기
⚠️
"SSO를 사용한 인증은 익명의 로그인 보안 위협이 발생한다" ❌ → 오답!
SSO는 통합인증 서버로 보안서버 역할을 수행한다. 하지만 익명의 로그인은 허용되지 않기 때문에 보안 위협이 발생하지 않는다.

SSO의 올바른 보안 위협:
① 블록킹 — 다수를 대상으로 하는 네트워크상에서 위장에 대한 위협이 발생할 수 있다 ✅
② 정당한 사용자만이 인증 서버와 응용 서버에게 자신이 정당한 사용자임을 증명할 수 있어야 한다 ✅
④ 사용자 인증 및 서버 인증을 위해 인증을 수행할 때 인증정보, 인증토큰, 토큰 ID 등의 다양한 인증정보가 저장 및 전송된다. 이때 인증정보 노출 보안 위협이 존재할 수 있다 ✅

대표적인 SSO 보안 위협: 위장 위협, 인증정보 노출, 인증정보 재사용, 기관리 위협, 세션 관리 위험
XSS 취약점 코드 판별 필기+실기
🔍
JAVA Servlet 코드에서 getParameter 함수는 HTML에서 전달되고 입력 값을 id 변수에 저장한다. 그리고 Name 부분은 JAVA JSP는 입력 값을 받아서 id 변수에 저장한다. 그리고 코드의 id 값을 HTML로 출력하는 것이다 → XSS 취약점

코드 패턴: request.getParameter("id") → Name: <%=id%>
사용자 입력값을 그대로 HTML에 출력 → XSS(Cross-Site Scripting) 취약점
