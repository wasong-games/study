Chapter 01

컴퓨터가 어떻게 작동하는가
— CPU, 레지스터, 메모리 계층

시스템 보안을 이해하려면 먼저 컴퓨터가 프로그램을 어떻게 실행하는지 알아야 해. CPU는 메모리에서 명령어를 하나씩 꺼내서 실행하는데, 이때 "지금 어디를 실행 중인지", "결과값을 어디 담는지"를 기억하는 작은 임시 저장소가 레지스터야. 보안 공격의 상당수가 이 레지스터 값을 조작하는 것을 목표로 한다는 걸 먼저 기억해둬.
레지스터 필기

레지스터는 CPU 안에 있는 초고속 임시 저장소야. 일반 메모리(RAM)보다 100배 이상 빠르지만 크기는 아주 작아. 64비트 운영체제 기준으로 주요 레지스터 이름 앞에 R이 붙고, 32비트는 E가 붙어.

64비트32비트이름의 의미용도 (왜 이 이름인가)
RAXEAXAccumulator 함수의 반환값(return value)을 담는 곳. 함수가 끝나고 나서 결과를 꺼내려면 RAX를 봐야 해.
RBXEBXBase 메모리 주소를 저장할 때 기준점(Base)으로 씀. 데이터 구조 접근 시 활용.
RCXECXCounter 반복문 카운터. for 루프가 몇 번 돌았는지 세는 용도. LOOP 명령어가 RCX를 자동으로 감소시켜.
RBPEBPBase Pointer 현재 스택 프레임의 바닥 주소. 함수 호출 시 이전 스택 상태를 복원하는 기준점 역할.
RSPESPStack Pointer 스택의 현재 꼭대기 주소. push/pop 할 때마다 RSP가 위아래로 움직여.
RIPEIPInstruction Pointer 다음에 실행할 명령어의 주소. CPU는 이 주소를 보고 다음 명령어를 가져와. 버퍼 오버플로우 공격의 핵심 타깃 — 이 값을 덮으면 원하는 코드를 실행시킬 수 있어.
RFLAGSEFLAGSFlags 연산 결과 상태를 비트로 저장. Zero Flag(결과=0), Carry Flag(올림), Sign Flag(음수) 등. 조건 분기(JE, JNE)가 이걸 보고 판단해.
🎯
버퍼 오버플로우 공격의 목표 = RIP(EIP)를 덮어쓰기
스택에 데이터를 너무 많이 쓰면 → RBP를 넘어서 → RIP(리턴 주소)까지 덮어써 → CPU가 공격자가 원하는 코드로 점프.
이래서 RIP가 시스템 보안에서 가장 중요한 레지스터야.
메모리 계층구조 필기

컴퓨터는 여러 종류의 저장소를 계층적으로 사용해. 핵심 트레이드오프는 하나야: 빠를수록 비싸고 작다. 느릴수록 싸고 크다. CPU와 가까울수록 빠른 저장소를 쓰고, 멀어질수록 느리지만 큰 저장소를 써.

왜 이 계층이 보안에 중요한가?
가상 메모리, 스왑(Swap), 버퍼 오버플로우, 메모리 포렌식 모두 이 계층 개념 위에서 동작해. 예를 들어 RAM에 있는 데이터는 전원이 꺼지면 사라지지만 (휘발성), SSD에 기록된 건 남아있어 — 포렌식에서 중요한 포인트야.
Chapter 02

OS의 핵심 역할
— 프로세스 관리와 스케줄링

운영체제(OS)의 가장 중요한 역할은 여러 프로그램이 동시에 실행되는 것처럼 보이게 하는 거야. 실제로 CPU는 한 번에 하나의 작업만 처리하는데, OS가 아주 빠르게 여러 프로세스를 번갈아 실행하면서 동시에 돌아가는 것처럼 착각하게 만들어. 이 과정에서 "누구한테 CPU를 줄지"를 결정하는 게 스케줄링이야.
프로세스 상태와 문맥 교환 필기

프로세스는 실행 중에 여러 상태를 오가며 전이해. 이 상태 변화를 이해하면 스케줄링도 자연스럽게 이해가 돼.

🔄
문맥 교환 (Context Switching)이란?
CPU가 프로세스 A를 멈추고 프로세스 B를 실행하려면, A의 현재 상태(레지스터 값들, PC, 스택 정보 등)를 PCB(Process Control Block)에 저장해야 해. 그래야 나중에 A를 다시 실행할 때 "아, 여기까지 했었지" 하고 이어서 할 수 있거든.
이 "저장하고 → B의 상태를 꺼내서 → CPU에 올리는" 과정이 문맥 교환이야.
문맥 교환이 자주 일어날수록 CPU 오버헤드가 커져 — 스케줄링을 잘 해야 하는 이유.
CPU 스케줄링 기법 필기

스케줄링은 크게 두 가지로 나뉘어. 비선점형은 일단 CPU를 잡으면 스스로 반납하기 전까지 뺏지 않는 방식이고, 선점형은 OS가 강제로 CPU를 빼앗을 수 있어. 선점형이 더 응답성이 좋지만 문맥 교환 오버헤드가 있어.

비선점형 (Non-Preemptive) — "일단 잡으면 끝까지"

  • FCFS (First Come First Served)
    대기 큐 도착 순서대로. 구현이 제일 단순. 단점: 짧은 작업이 긴 작업 뒤에 오면 오래 기다려야 함 — 이걸 Convoy Effect라고 해.
  • SJF (Shortest Job First)
    수행 시간이 가장 짧은 작업 먼저. 평균 대기시간은 이론상 최적. 단점: 짧은 작업이 계속 들어오면 긴 작업이 영원히 못 실행될 수 있어 — 이걸 기아(Starvation)라고 해.
  • HRN (Highest Response Ratio Next)
    SJF의 기아 문제 보완. 우선순위 = (대기시간 + 서비스시간) / 서비스시간. 오래 기다릴수록 우선순위가 올라가.

선점형 (Preemptive) — "OS가 강제로 뺏을 수 있다"

  • Round Robin (RR)
    FCFS처럼 순서대로 실행하되, 각 프로세스에게 Time Quantum(시간 할당량)만큼만 줘. 시간이 지나면 강제로 뺏음. 대화형 시스템에 적합. Time Quantum이 너무 크면 = FCFS, 너무 작으면 = Context Switching만 반복하는 낭비.
  • SRT (Shortest Remaining Time)
    SJF의 선점형 버전. 새 프로세스가 도착할 때마다 "지금 실행 중인 것과 신입 중 누가 더 빨리 끝나나?" 비교해서 더 짧은 것 실행.
  • Multi Level Feedback Queue
    여러 큐를 두고 우선순위 큐 + Round Robin을 결합한 하이브리드. 실제 현대 OS(Linux, Windows)가 이 방식 사용.
교착상태 (Deadlock) 필기

교착상태는 두 개 이상의 프로세스가 서로 상대방이 가진 자원을 기다리며 영원히 멈춰버린 상태야. 예를 들어 A는 프린터를 잡고 스캐너를 기다리고, B는 스캐너를 잡고 프린터를 기다리면 — 둘 다 영원히 기다려. 이게 교착상태.

🔒
교착상태 발생 4조건 — 4개가 동시에 성립해야 발생 (하나라도 깨면 예방 가능)

① 상호배제 (Mutual Exclusion) — 자원은 한 번에 하나의 프로세스만 사용 가능.
→ 해결: 자원을 공유 가능하게 만들기 (현실적으로 어려운 경우 많음)

② 점유와 대기 (Hold and Wait) — 이미 자원을 갖고 있으면서 추가 자원을 요청하며 대기.
→ 해결: 필요한 자원을 한꺼번에 모두 할당하고 시작하기

③ 비선점 (Non-Preemption) — 프로세스가 점유한 자원을 강제로 빼앗을 수 없음.
→ 해결: 선점 허용 (자원을 강제로 회수)

④ 환형 대기 (Circular Wait) — A→B→C→A처럼 자원 요청이 원형 고리 형성.
→ 해결: 자원에 번호를 매기고 낮은 번호 순서로만 요청 가능하게
🧠 암기법
상점비환 — 상호배제, 점유와대기, 비선점, 환형대기

Chapter 03

메모리가 부족할 때
— 가상 메모리와 교체 알고리즘

RAM이 4GB인데 10GB짜리 프로그램을 실행하려면 어떻게 해야 할까? OS는 가상 메모리 기법을 써서 마치 메모리가 훨씬 많은 것처럼 흉내내. SSD/HDD의 일부를 RAM처럼 사용하는 거야 — 느리지만, 없는 것보다는 낫지. 하지만 그만큼 "어떤 페이지를 메모리에 유지하고 어떤 걸 디스크로 내보낼지" 결정하는 게 중요해.
가상 기억장치 관리 기법 4종류 필기
종류결정하는 것대표 기법
할당 기법
(Allocation)		 프로세스에게 메모리를 어떤 단위로 줄지 고정 할당, 가변 할당, Paging (고정 크기 페이지), Segmentation (논리적 단위)
호출 기법
(Fetch Policy)		 디스크 → RAM으로 언제 불러올지 Demand Fetch (필요할 때만 = 지금 대부분 OS 방식), Pre Fetch (미리 예측해서 로드)
배치 기법
(Placement)		 불러온 페이지를 RAM 어디에 놓을지 First Fit (처음 맞는 공간), Best Fit (딱 맞는 공간), Worst Fit (제일 큰 공간)
교체 기법
(Replacement)		 RAM이 꽉 찼을 때 어떤 페이지를 내보낼지 FIFO, LRU, LFU, NUR, SCR, Optimal 등
교체 기법 상세 — 어떤 페이지를 내쫓을 것인가 필기

교체 기법은 직관적으로 생각해봐. "앞으로도 안 쓸 것 같은 페이지"를 내쫓아야 성능이 좋아져. 근데 미래를 알 수 없으니, 각 알고리즘마다 다른 근사 방법을 써.

알고리즘기준 (뭘 내쫓나)특징 및 단점
Random 무작위로 선택 오버헤드 없음. 성능 보장 안 됨.
FIFO
(First In First Out)		 가장 먼저 올라온 페이지를 먼저 내보냄 구현 간단. 단, 오래됐다고 안 쓰는 게 아닐 수 있어. Belady의 이상: 프레임을 늘렸는데 오히려 페이지 결함이 늘어나는 현상 발생 가능.
LFU
(Least Frequently Used)		 사용 횟수가 가장 적은 페이지 "가끔 쓰이지만 중요한" 페이지보다 최근에 막 올라온 페이지가 먼저 쫓겨날 수 있음.
LRU
(Least Recently Used)		 가장 오랫동안 사용되지 않은 페이지 현실에서 가장 많이 쓰이는 근사 알고리즘. "최근에 안 썼으면 앞으로도 안 쓸 가능성 높다"는 시간 지역성 이용. 단, 타임스탬프 유지 오버헤드 있음.
Optimal 앞으로 가장 오랫동안 사용되지 않을 페이지 이론상 최적이지만 미래를 알아야 해서 실현 불가능. 성능 비교의 기준(벤치마크)으로만 사용.
NUR
(Not Used Recently)		 참조 비트(0/1)와 수정 비트(0/1)로 판단 LRU의 오버헤드를 줄인 근사 버전. 비트 조합 우선순위: (0,0) → (0,1) → (1,0) → (1,1) 순으로 교체.
SCR
(Second Chance)		 FIFO 기반. 참조 비트가 1이면 0으로 바꾸고 한 번 더 기회를 줌 FIFO의 단점 보완. 참조된 적 있으면 한 번 더 살아남을 기회. Clock 알고리즘이라고도 불림.
💡
스래싱 (Thrashing)이란?
메모리가 너무 부족해서 페이지 교체가 너무 자주 일어나면, CPU가 실제 작업보다 페이지 교체에 더 많은 시간을 쓰게 돼. 이 상태를 스래싱이라고 해. 해결책은 프로세스 수를 줄이거나 메모리를 늘리는 것.
Chapter 04

디스크와 RAID
— 데이터를 어떻게 저장하고 보호하는가

디스크는 HDD의 경우 물리적으로 헤드가 움직여서 데이터를 읽어. "헤드를 어떤 순서로 움직일 것인가"가 디스크 스케줄링이야. RAID는 여러 개의 디스크를 묶어서 성능이나 안정성을 높이는 기술이야 — 보안에서는 데이터 가용성(Availability)과 직결되는 개념이야.
디스크 스케줄링 기법 필기
알고리즘방식비유 / 핵심 특징
FCFS 도착 순서대로 처리 은행 번호표처럼 순서대로. 공평하지만 헤드가 앞뒤로 많이 왔다 갔다 해서 비효율적.
SSTF
(Shortest Seek Time First)		 현재 헤드 위치에서 가장 가까운 트랙 먼저 처리량이 높음. 단, 헤드와 멀리 있는 요청은 계속 밀려나는 기아 현상 발생 가능.
SCAN
(엘리베이터 알고리즘)		 헤드가 한 방향으로 쭉 이동하며 처리, 끝에 도달하면 역방향 엘리베이터처럼 한 방향으로 쭉. SSTF보다 공평. 단, 양 끝 쪽은 한 번만 지나쳐서 중간보다 대기 시간 길 수 있음.
C-SCAN
(Circular SCAN)		 한 방향으로만 이동. 끝에 도달하면 처음 위치로 즉시 이동 후 다시 시작 SCAN보다 대기 시간 균등. 돌아올 때 요청 처리 안 함 — 대신 더 균일한 서비스.
C-LOOK C-SCAN과 같지만 디스크 맨 끝까지 안 가고 마지막 요청까지만 처리 후 복귀 C-SCAN에서 불필요한 이동 제거. 실제로 가장 효율적인 방식 중 하나.
RAID (Redundant Array of Independent Disks) 필기

RAID는 "여러 개의 디스크를 하나처럼 쓰는 기술"이야. 목적은 두 가지: 성능 향상(여러 디스크에 나눠 쓰면 빠름)과 안정성 향상(복사본이나 패리티를 두면 하나 죽어도 살아남음).

레벨방식최소 디스크복구 가능?요약
RAID 0Striping — 데이터를 여러 디스크에 분산 저장 2개불가 속도는 제일 빠름. 하지만 하나라도 고장나면 데이터 전체 손실. 백업 없음.
RAID 1Mirroring — 두 디스크에 똑같이 복사 2개가능 (1개 고장) 안정성 최高. 비용 2배. 읽기 빠름, 쓰기는 동일. 작은 서버의 OS 디스크에 많이 씀.
RAID 2Hamming Code ECC — 비트 단위 오류 정정 3개 이상가능 현재 거의 미사용. ECC 메모리 자체에 내장된 경우 많아.
RAID 3Parity를 별도 디스크에 저장 (Byte 단위 I/O) 3개가능 (1개 고장) 패리티 디스크가 병목. RAID 5에 밀려 현재 잘 안 씀.
RAID 4Parity를 별도 디스크에 저장 (Block 단위 I/O) 3개가능 (1개 고장) RAID 3의 Block 버전. 마찬가지로 패리티 디스크 병목.
RAID 5Parity를 여러 디스크에 분산 저장 3개 (일반적으로 4개)가능 (1개 고장) 가장 많이 쓰이는 RAID. 성능 + 안정성의 균형. 패리티 분산으로 병목 없음.
RAID 6이중 Parity 분산 저장 4개가능 (2개 동시 고장) RAID 5보다 안전. 쓰기 성능은 약간 떨어짐. 대용량 스토리지에 사용.
RAID 10
(1+0)		RAID 1 (미러링) + RAID 0 (스트라이핑) 결합 4개가능 (미러 쌍 1개 고장) 성능 + 안정성 모두 좋음. 비용도 많이 듦. 고성능 DB 서버에 주로 사용.
💡
RAID와 보안(CIA 3요소)
RAID는 가용성(Availability)을 높이는 기술이야. 디스크가 고장나도 서비스를 계속 제공할 수 있게 해주거든. 단, RAID는 백업이 아니야. 랜섬웨어가 파일을 암호화하면 RAID도 함께 암호화된 상태로 복제돼. 백업은 별도로 오프사이트 또는 오프라인으로 해야 해.
Chapter 05

리눅스
— 부팅부터 인증, 권한, 로그까지

정보보안기사 실기에서 리눅스 관련 문제가 굉장히 많이 나와. 특히 /etc/passwd, /etc/shadow 파일 구조, 파일 권한 표기법, cron 설정, 로그 파일 종류는 거의 매년 출제돼. 실제로 시스템을 운영하면서 자연스럽게 익히는 게 제일 좋지만, 시험 준비용으로 흐름을 잡아줄게.
리눅스 부팅 순서 필기

컴퓨터 전원을 켜면 어떤 일이 일어나는지 순서대로 알면 부팅 이슈나 침해 사고 시 원인 파악이 쉬워.

① BIOS / UEFI
전원 ON → ROM에 저장된 BIOS 코드 실행 → 하드웨어 초기화(POST) → 부팅 디스크의 MBR(Master Boot Record)을 찾아.
→
② 부트로더
(GRUB/LILO)
MBR에서 부트로더 실행. 보조 기억장치에 저장된 리눅스 커널(Linux Kernel)을 찾아서 RAM에 올림. 현재 대부분 GRUB2 사용.
→
③ 커널 초기화
커널이 RAM에 올라와서 하드웨어 드라이버 로드, 파일시스템 마운트 등 시스템 초기화.
→
④ init 프로세스
(PID = 1)
커널이 최초로 실행하는 프로세스. PID 번호는 항상 1. /etc/inittab 파일을 읽어 Run Level에 맞게 디바이스·프로세스를 활성화. 현재 대부분 systemd가 init을 대체.
Run Level설명언제 씀?
0시스템 종료 (Halt)init 0 = 전원 끄기
1단일 사용자 모드 (Single User)비밀번호 분실 복구, 긴급 수리 시
2다중 사용자 (NFS 없음)네트워크 파일 공유 없는 멀티유저
3다중 사용자 + 네트워크 (텍스트 모드)일반 서버 — 그래픽 없이 CLI만
5다중 사용자 + 네트워크 + X11 (그래픽)일반 데스크톱 / GUI 서버
6재부팅 (Reboot)init 6 = 재시작
리눅스 파일시스템 — EXT3 vs EXT4 필기

EXT3

  • 단일 파일 크기 제한: 4 GB
  • 파일명: 최대 256 Byte
  • 최대 파일시스템 크기: 16 TB
  • 디렉토리 당 최대 파일 수: 65,565개
  • 저널링(Journaling) 지원 — 파일 시스템 손상 시 복구 빠름

EXT4

  • 단일 파일 크기: 16 TB (EXT3의 4000배)
  • 최대 파일시스템: 1 EB (엑사바이트)
  • EXT3과 하위 호환 가능
  • Extents 기술 → 연속 블록 관리 효율화
  • 현재 리눅스 서버 기본 파일시스템
리눅스 인증 — /etc/passwd 파일 구조 필기+실기

/etc/passwd는 시스템에 등록된 모든 사용자 정보를 담은 파일이야. 과거엔 패스워드도 여기에 저장했는데, 보안 문제로 지금은 패스워드를 /etc/shadow로 분리했어.

# /etc/passwd 파일 형식 (콜론으로 7개 필드 구분) root : x : 0 : 0 : root : /root : /bin/bash (1) (2) (3) (4) (5) (6) (7)
번호필드명설명
(1)Login Name사용자 계정명. 로그인 시 입력하는 이름.
(2)Passwordx로 표시되면 패스워드가 /etc/shadow에 저장됨. 예전엔 암호화된 패스워드가 여기 있었어.
(3)User ID (UID)사용자 고유 번호. root = 0. 일반 사용자는 1000번부터.
(4)Group ID (GID)기본 그룹 번호. root 그룹 = 0.
(5)Comments사용자 설명(이름, 연락처 등). chfn 명령어로 변경 가능.
(6)Home Directory로그인 시 이동하는 홈 디렉토리 경로.
(7)Shell사용자가 기본으로 사용하는 셸. /bin/bash, /bin/sh, /sbin/nologin 등. nologin이면 로그인 불가 계정.
리눅스 인증 — /etc/shadow 파일 구조 필기+실기

shadow 파일은 루트만 읽을 수 있어. 패스워드가 단방향 해시로 저장되어 있고, 패스워드 만료 정책도 여기서 관리해.

# /etc/shadow 파일 형식 (콜론으로 9개 필드) root : $6$salt$hashedpassword : 19000 : 0 : 99999 : 7 : : : (1) (2) (3) (4) (5) (6) (7)(8)(9)
번호필드명설명
(1)Login Name사용자 계정명
(2)Encrypted Password암호화된 패스워드. 앞의 $숫자$가 해시 알고리즘: $1$=MD5, $5$=SHA-256, $6$=SHA-512. $와 $ 사이 값이 Salt — 레인보우 테이블 공격 방어용.
(3)Last Changed1970년 1월 1일(유닉스 에포크)부터 패스워드가 마지막으로 바뀐 날까지의 일수.
(4)Minimum패스워드를 바꾼 후 최소한 며칠은 지나야 또 바꿀 수 있는지. 0이면 제한 없음.
(5)Maximum패스워드 최대 유효 기간(일). 99999면 거의 무제한. 180이면 180일마다 강제 변경.
(6)Warn만료 며칠 전부터 경고를 보여줄지. 7이면 만료 7일 전부터 "곧 만료됩니다" 메시지.
(7)Inactive패스워드 만료 후 며칠까지 로그인을 허용할지 (유예 기간).
(8)Expire계정 자체의 만료 날짜 (에포크 기준 일수). 이 날이 지나면 계정 전체 잠김.
(9)Reserved현재 미사용 (향후 확장용)

umask 계산법
umask는 새 파일/디렉토리 생성 시 빼줄 권한을 지정해.
• 파일 기본 권한: 666 - umask (실행 권한은 기본적으로 부여 안 함)
• 디렉토리 기본 권한: 777 - umask
umask가 022면 → 파일: 644, 디렉토리: 755
umask가 027면 → 파일: 640, 디렉토리: 750
리눅스 파일 권한 표기법 필기+실기

특수 권한 (setuid, setgid, sticky bit) 필기+실기

특수 권한은 일반 rwx 권한 외에 실행 시 파일 소유자의 권한으로 동작하게 하는 기능이야. 대표적인 예: /usr/bin/passwd — 일반 유저가 실행해도 root 권한으로 /etc/shadow를 수정할 수 있어. 이 기능이 잘못 설정되면 권한 상승 공격에 악용돼.

cron — 자동 실행 스케줄러 필기+실기

cron은 특정 시간에 명령어를 자동으로 실행하는 리눅스 스케줄러야. 공격자들이 지속성(Persistence) 유지를 위해 악성 스크립트를 crontab에 등록하기도 해서, 보안 점검 시 꼭 확인해야 해.

Chapter 07

시스템 공격의 원리
— 버퍼 오버플로우부터 킬체인까지

공격자가 어떻게 시스템을 뚫는지 원리를 알면 방어도 논리적으로 이해가 돼. 단순 암기가 아니라 "왜 이 공격이 가능하고, 왜 이 방어 기법이 효과적인지"를 연결해서 이해해봐.
버퍼 오버플로우 (Buffer Overflow) 필기+실기

버퍼(Buffer)는 임시로 데이터를 저장하는 메모리 공간이야. C언어는 배열이나 문자열을 다룰 때 경계를 검사하지 않는 함수들이 있어서, 할당된 크기보다 큰 데이터를 쓰면 인접한 메모리를 덮어쓰게 돼.

💥
스택 구조와 공격 원리
함수가 호출되면 스택에 이런 순서로 쌓여:

[함수 인자] → [리턴 주소(RIP)] → [이전 RBP] → [지역변수 버퍼]

공격자가 지역변수 버퍼에 충분히 큰 데이터를 쓰면 → RBP를 덮어쓰고 → 리턴 주소(RIP)까지 덮어써.
함수가 return할 때 CPU가 그 덮어쓴 주소로 점프 → 원하는 코드 실행.

취약한 C 함수 — 이 함수들은 입력 크기를 확인하지 않아:

# 취약한 함수 → 안전한 대체 함수 strcpy(dest, src) → strncpy(dest, src, n) strcat(dest, src) → strncat(dest, src, n) gets(buf) → fgets(buf, size, stdin) scanf("%s", buf) → scanf("%255s", buf) // 크기 제한 추가 sprintf(buf, fmt, ...) → snprintf(buf, size, fmt, ...) getwd(buf) → getcwd(buf, size)
방어 기법원리핵심 키워드
Stack Guard
(Canary)		 함수 진입 시 스택에 canary 값(랜덤 값)을 삽입. 함수 종료 시 canary가 변경됐으면 → 오버플로우 발생 → 프로그램 종료. 카나리아 새처럼 먼저 죽어서 경보를 울림
Stack Shield 함수 호출 시 리턴 주소를 별도의 Global RET 스택에 복사. 리턴 시 두 값을 비교해서 변조 여부 확인. 리턴 주소의 백업본을 별도 보관
ASLR
(Address Space Layout Randomization)		 스택·힙·라이브러리의 시작 주소를 실행할 때마다 랜덤으로 배치. 공격자가 리턴 주소를 예측하기 어렵게. 주소를 매번 바꿔서 예측 불가
DEP / NX bit
(Data Execution Prevention)		 스택·힙 영역에 실행 권한을 제거. 데이터 영역에서 코드가 실행 안 됨. 하드웨어 NX 비트 활용. 데이터는 실행 못 한다
NOP Sled NOP(No Operation) 명령어를 길게 나열해서 그 뒤에 쉘코드 배치. ASLR이 없던 시절의 공격 기법. 탐지법: NOP 패턴(0x90) 연속 탐지. 미끄럼틀처럼 NOP를 타고 내려와 쉘코드 실행
Relro
(RELocation Read-Only)		 ELF 바이너리의 GOT(Global Offset Table) 등 데이터 영역을 Read-Only로 설정. GOT 덮어쓰기 공격 방어. Write 못 하게 잠금
PIE
(Position Independent Executable)		 코드 영역까지 ASLR을 적용. ASLR은 스택·힙만 랜덤인데, PIE는 코드 영역까지 랜덤. ASLR의 완전체 버전
💡
DEP/ASLR 우회 기법 (시험에 자주 나옴)
• Return to libc: DEP로 스택 실행이 안 될 때, 이미 메모리에 있는 libc의 system() 함수 주소로 점프하는 기법.
• ROP (Return Oriented Programming): 코드 곳곳에 있는 짧은 코드 조각(가젯, gadget)을 체인으로 연결해서 원하는 동작 수행. ASLR+DEP 동시 우회 가능.
사이버 킬체인 (Cyber Kill Chain) 필기+실기

Lockheed Martin이 제안한 모델로, APT(지능형 지속 공격) 공격자가 어떤 단계로 침투하는지 설명해. 각 단계에서 공격을 탐지하고 차단하는 게 목표야.

단계활동명공격자가 하는 일방어 관점
1단계정찰 (Reconnaissance) 목표 조직의 IT 인프라, 직원 정보, 기술 스택 등을 수집. 공개 정보(OSINT), 소셜 엔지니어링 활용. 공개 정보 최소화, 직원 보안 교육
2단계무기화 및 전달 (Weaponization) 수집한 정보를 바탕으로 공격용 도구(익스플로잇, 악성 문서 등) 제작. 피싱 이메일이나 드라이브-바이 다운로드로 전달. 이메일 필터링, 샌드박스 분석
3단계익스플로잇 설치 (Exploitation) 취약점을 이용해 악성코드를 실행. 드로퍼(Dropper)로 본격 악성코드 설치. 패치 관리, 취약점 스캔
4단계명령 및 제어 (C2/C&C) 감염 시스템과 공격자 서버 간 C2 채널 수립. 원격으로 명령 실행. C2 트래픽 탐지, DNS 모니터링
5단계행동 및 탈출 (Actions on Objectives) 목표 달성: 데이터 유출, 랜섬웨어 배포, 시스템 파괴. 이후 로그 삭제 등으로 흔적 제거. DLP, 이상 행위 탐지, 포렌식
주요 시스템 공격 기법 필기
공격 기법원리
레이스 컨디션
(Race Condition)		 두 프로세스가 동시에 같은 자원에 접근할 때 발생하는 경쟁 조건. 예: setuid된 프로그램이 임시 파일을 사용할 때, 그 파일을 공격자 제어 파일로 심볼릭 링크 교체 → root 권한으로 원하는 파일에 쓰기 가능.
포맷 스트링 공격
(Format String)		 printf(user_input)처럼 형식 문자열이 사용자 입력이면, %x %x %n 등을 삽입해서 스택 메모리 읽기/쓰기 가능. %n은 지금까지 출력한 바이트 수를 메모리에 씀 — 이 특성으로 임의 메모리 덮어쓰기 가능.
Use After Free (UAF) 메모리를 free()로 해제한 뒤에도 그 포인터를 계속 사용하는 버그. 해제된 메모리를 공격자가 원하는 데이터로 채우면 → 해제된 포인터로 공격자 데이터 실행 가능.
셸쇼크 (Shell Shock) GNU Bash Shell의 환경변수 처리 취약점. 환경변수에 함수 정의 후 세미콜론 뒤에 명령어를 붙이면 실행됨. CGI, DHCP 클라이언트 등을 통해 원격 코드 실행 가능. 과학기술정보통신부와 KISA가 발표.
패스워드 크래킹 • 사전 공격: 단어 목록 대입
• 무차별 대입(Brute Force): 모든 조합 시도
• 레인보우 테이블: 해시 역산 테이블 사용 — Salt로 방어
• Pass the Hash: NTLM 해시 자체를 인증 토큰으로 재사용 (해시 탈취 후 패스워드 불필요)

악성코드와 인증 위임
— 바이러스 세대, OAuth

바이러스 세대 분류 필기

바이러스는 백신 소프트웨어의 발전에 맞서 진화해왔어. 1세대부터 5세대까지 알아두면 이후 나오는 악성코드 대응 방식과도 연결돼.

세대유형특징백신 대응법
1세대원시형 바이러스 코드 구조가 단순하고 분석이 쉬움. 자기복제 후 파일에 부착. 시그니처 기반 탐지 가능
2세대암호화형 바이러스 바이러스 본체를 암호화해서 시그니처 탐지 회피. 실행 시 복호화 루틴이 먼저 실행됨. 복호화 루틴 시그니처로 탐지
3세대은폐형 바이러스 메모리에 상주하면서 감염 사실을 숨김. 감염 파일의 크기가 증가하지 않는 것처럼 보이게 조작. 감염 파일 읽으면 원본 내용을 보여줌. 메모리 상주 탐지, 부팅 시 스캔
4세대갑옷형 바이러스 암호화 + 다양한 안티 디버깅 기법 결합. 분석 도구가 붙으면 다르게 동작하는 등 역분석 방해. 동적 분석, 샌드박스
5세대매크로형 바이러스 운영체제와 무관하게 MS Office 같은 응용 프로그램 내 매크로 언어로 동작. 문서 파일로 전파. 매크로 실행 비활성화, 이메일 필터링
🦠
기타 주요 악성코드 유형
• 웜(Worm): 숙주 파일 없이 스스로 복제·전파. 네트워크를 통해 빠르게 퍼짐 (WannaCry가 대표적)
• 트로이 목마: 정상 프로그램인 척하지만 악성 기능 숨김. 자기 복제 없음.
• 랜섬웨어: 파일을 암호화 후 복구 대가로 금전(가상화폐) 요구
• 스파이웨어: 사용자 정보(키로깅, 스크린샷)를 몰래 수집·전송
• 루트킷(Rootkit): 자신을 숨기면서 시스템에 지속적으로 상주. 관리자 권한 유지.
• APT (Advanced Persistent Threat): 특정 조직을 목표로 장기간 은밀하게 지속되는 공격
OAuth 1.0 vs 2.0 필기+실기

OAuth는 "내 SNS 계정으로 다른 서비스에 로그인하기" 같은 인증 위임 프로토콜이야. "구글로 로그인" 버튼을 누르면 OAuth가 동작하는 거야. 핵심은 패스워드를 서드파티에 주지 않고 인증할 수 있다는 거야.

inode와 포렌식 타임스탬프 필기+실기

리눅스 포렌식에서 inode는 빠질 수 없는 개념이야. 파일에 대한 메타데이터가 담겨 있어서 침해 사고 시 공격자가 무슨 일을 했는지 파악하는 데 핵심 역할을 해.

📁
inode에 저장되는 것: 파일 크기, 소유자(UID/GID), 권한, 타임스탬프, 파일 유형, 하드링크 수, 데이터 블록 포인터
inode에 없는 것: 파일명 — 파일명은 디렉토리에 저장됨. 같은 inode를 여러 이름이 가리키는 게 하드링크.

타임스탬프 3종류 (MAC Time) — 포렌식 핵심
• atime (Access Time): 마지막으로 읽은(접근한) 시간
• mtime (Modify Time): 마지막으로 내용을 수정한 시간
• ctime (Change Time): 마지막으로 속성(권한·소유자)이 변경된 시간

공격자가 파일 내용을 수정하면 mtime이 바뀌지만, 소유자나 권한을 바꾸면 ctime이 바뀌어. touch -t로 atime·mtime은 조작 가능하지만, ctime은 조작하기 어려워 — 포렌식에서 가장 신뢰하는 타임스탬프.
